# Multi-Agent Development System

## [最重要] 必須で守る実装方向性と設計原則

- 公式推奨・標準準拠：余計な抽象を持ち込まないシンプルなロジックで“根本から賢く”実装
- 誤った抽象は延命しない：歴史的負債や誤抽象はその場で削除し、必要なら正しい抽象を再導入する。フォールバック的延命より「消して作り直す」
- 失敗は隠さない：問題は即エラーで顕在化。フォールバックではなくエラー、マイグレーションではなく完全リセットで対応
- 完全ログ主義：UI を含むすべての動作確認をログだけでリプレイ可能な情報量を確保。
  - 可観測性ファースト
    - 入力（リクエスト/環境/設定）
    - 処理（分岐/外部 I/O/リトライ）
    - 出力（結果/副作用）
    - すべてを追跡可能にする。タイムスタンプ・所要時間は必須。
  - ログレベル定義
    - INFO：前提や環境情報
    - DEBUG・STATES：詳細なデバッグ情報。変数の値、内部状態など
    - SUCCRSS：正常な動作の記録。ツール呼出開始・終了、ファイル読書など
    - WARNING：警告（処理は継続）。非推奨 API の使用、パフォーマンス低下など
    - ERROR：粒度を問わず全てのエラーが対象

## 前提

- 回答は日本語。
- ファイル読込は最大行数で広範囲に取得。断片的読込を避ける
- Markdown ファイルを外部コンテクストウィンドウとして活用
- パッケージ・ライブラリは全て最新を使うためインストール時はバージョン指定不要
- `coding-confidant` を使っても求めたレベルの回答が得られなければ、送信文脈や task の切り方を変えて再度調査する
- 最新のフレームワーク動向やコミュニティ反応が重要な依頼では、`coding-confidant` と `trend-researcher` を並行で動かす

## 外部 LLM / MCP 送信の扱い

- `coding-confidant` など外部送信を伴う調査・相談では、「ローカルで読んだだけ」と見なさない
  - 送信物の実体を常に明示する
    - ベース prompt
    - `--task` / `--task-file`
    - repomix 出力
    - include した `_docs`、ログ、設定、補助ファイル
- 外部送信前に文脈を整理する
  - `--include` / `--ignore`、または同等の手段で不要物を除外したうえで、コンテクスト上限まで必要十分な情報を含める
  - 認証情報、トークン、秘密鍵、ローカル状態、キャッシュ、`node_modules`、`.venv`、明らかな一時物は原則送らない
  - ログ、生成物、調査メモは「不要だから除外」ではなく、依頼達成に効くなら積極的に含める
- 送信対象に秘密情報や認証状態が混ざる恐れがあり、除外を確認できない場合は停止して依頼主に確認する
- 外部送信した request / response は再確認可能な形で保存する
  - これは prompt で指示する話ではなく、skill / script 側の責務として実装する
  - `coding-confidant` では `<repo>/_docs/_skills/coding-confidant/<timestamp-pid>/openai.jsonl` と repomix ファイルを保存物の正本とする
  - `trend-researcher` では `<repo>/_docs/_skills/trend-researcher/<timestamp-pid>/x-search.json`, `x-search.jsonl`, `openai.jsonl`, `report.md` を保存物の正本とする
  - 送信後の判断やレビューは、保存済みの raw データを根拠に行う

## 対応概要

1. プロジェクトのファイルツリーをコマンドで確認
2. _docs 配下を確認して過去に行った情報収集・実装を詳細に把握
3. Serena を用いてプロジェクト構造を把握
4. ライブラリ仕様を一次情報確認
  1. **Next.js の場合**: Next.js MCP で確認
  2. **GitHub 上の**関連パッケージ・ライブラリ: DeepWiki で確認
  3. **その他の**関連パッケージ・ライブラリ: Context7 で確認
  4. **上記で不足する場合**: `coding-confidant` で詳細具体に調査。X 上の最新動向や空気感が重要なら `trend-researcher` も並行で動かす
  5. **必要に応じて、**足りない情報を依頼主に確認
5. 収集した情報を元にdocs フォルダ配下にわかりやすいファイル名で Markdown ファイルを作成
6. 調査の過程で気がついた課題を記録・修正
  1. 提供されたログや実装から下記の疑い箇所を全てリストアップ
    1. 不用意/一時的/冗長/暫定/不要な要素の気配：
      1. 定義、関数、フラグ、ガード、リトライ、分岐、判定、フォールバック、ファイル、ディレクトリ
      2. 失敗の不可視化、黙殺、握りつぶし
    2. 自動復旧や「なあなあの継続」の気配。
    3. サービス体験向上の可能性がある追加要件
  2. 5-1-1, 5-1-2 の該当箇所を全て削除。代替が必要なら正しい抽象で再設計。
  3. 5-1-3 の該当箇所はdocs/issue.md に記載。
7. 依頼の実装設計
  - `coding-confidant` に下記を具体詳細に高密度な長文で渡して調査する
    - ライブラリ仕様を含めた当プロジェクトの現状
    - 依頼内容
    - 本ドキュメントの「必須で守る実装方向性と設計原則」
  - X 上の最新トレンド、採用感、反応速度が判断に効くテーマでは `trend-researcher` を並行で走らせる
8. 実装
  - 依頼を遂行する最もモダンで公式推奨なシンプルなロジックでスマートな根本から実装
  - 修正依頼時は以下のプロセスを反復して根治
  1. ログと実装からあり得る原因を全列挙
  2. 原因特定用のログを増設（計測点・相関 ID・入力/出力・リトライ理由など）
  3. 解決オプションを `coding-confidant` でリサーチ依頼（ライブラリ固有性・モダン性を考慮）。X 上の最新動向が効く場合は `trend-researcher` も並行で使う
  4. 全て選択肢を検証（ビルド・動作確認・ログ検査）
    - ブラウザで確認可能な場合：Chrome MCP  でコンソール/ネットワーク/トレース確認
    - Swift の場合は
  5. 結果をコメントとしてコード内にコメントアウトとして記録し、ベストプラクティスを決定（後からコード見ただけ経緯がわかるように）
  6. `coderabbit -c ~/.codex/AGENTS.md --plain`を使ってコードレビュー
  7. 依頼目的達成した上でコードレビューで問題が指摘されなくなるまで繰り返す
